================================================================================
1. 概要
================================================================================

本ガイドラインは、エスペラントの文章をすべて語根分解し、2文字以上の各語根に
日本語または韓国語のルビ注釈を付したHTML文書を、生成AIに修正させる際の
参考資料である。

修正作業の主なカテゴリ：
  - rt注釈テキストの語義の誤り修正
  - ルビ未付与の単語への注釈追加
  - CSSクラスの適正化（ピクセル幅比率ベース）
  - 同一語根に対する注釈の統一
  - 改行位置（<br>）の整合
  - HTML構造上の問題の修正

【必要ファイル（同一ディレクトリに配置すること）】
  (1) Unicode_BMP全范围文字幅(宽)_Arial16.json
      → 各Unicode文字の Arial 16px におけるピクセル幅データ
      → CSSクラス決定に必須（§2.3 で詳述）
  (2) ruby_css_verifier.py
      → CSSクラスの一括検証・自動修正スクリプト（§2.3 のアルゴリズム準拠）
      → 使い方: python3 ruby_css_verifier.py *.html --fix --margin 0.05
      → 境界ケース（ratio が閾値から0.05以内）をスキップする安全モード推奨

────────────────────────────────────────────────────────────────────────────────
2.1 CSSクラス一覧（8段階）
────────────────────────────────────────────────────────────────────────────────

本文書で使用するルビのCSSクラスは以下の8段階である。修正作業の前に、
この体系を正確に理解しておく必要がある。

  ┌──────────┬──────────┬─────────────┬──────────────────────┐
  │ クラス名  │フォント  │ margin-top  │ 用途                  │
  │           │ サイズ   │             │                       │
  ├──────────┼──────────┼─────────────┼──────────────────────┤
  │ XXXS_S   │ 0.3em    │ -8.3em      │ 3行注釈（<br>×2）     │
  │ XXS_S    │ 0.3em    │ -7.2em      │ 2行注釈（<br>×1）     │
  │ XS_S     │ 0.3em    │ -6.1em      │ 1行・極小フォント     │
  │ S_S      │ 0.4em    │ -4.85em     │ 1行・小フォント       │
  │ M_M      │ 0.5em    │ -4.00em     │ 1行・中フォント       │
  │ L_L      │ 0.6em    │ -3.55em     │ 1行・やや大フォント   │
  │ XL_L     │ 0.7em    │ -3.20em     │ 1行・大フォント       │
  │ XXL_L    │ 0.8em    │ -2.80em     │ 1行・最大フォント     │
  └──────────┴──────────┴─────────────┴──────────────────────┘

────────────────────────────────────────────────────────────────────────────────
2.2 基本ルール
────────────────────────────────────────────────────────────────────────────────

【<br>個数とクラスの対応（絶対厳守）】
  (a) <br>が2つ含まれる注釈（3行） → 必ず XXXS_S を使用
  (b) <br>が1つ含まれる注釈（2行） → 必ず XXS_S を使用
  (c) <br>を含まない注釈（1行）    → XS_S～XXL_L のいずれかを使用
  (d) XXXS_SやXXS_Sを1行注釈に使うと、margin-topが過大となり
      ルビが本文テキストの上方に浮いてしまうため、絶対に避ける。

【★★★ AIによる既存CSSクラスの手動変更禁止 ★★★】
  (e) AIが文字数比率等の近似でCSSクラスを手動再計算し、既存クラスを
      変更することは禁止する。近似計算ではピクセル幅ベースの正確な値に
      ならないため。
  (e2) ただし、注釈テキスト（rt）の語義を修正する場合（§3.1参照）は、
       修正後の (rb, 新rt) は新規ペアとなる。このとき：
       ・同一の (rb, 新rt) がファイル内に既存 → そのCSSクラスを使用 …(f)
       ・ファイル内に前例なし → §2.3 のアルゴリズムで再計算する …(g)
       この場合のCSSクラス変更は (e) の禁止対象には該当しない。
  (e3) ruby_css_verifier.py による一括検証・修正は推奨する。
       このスクリプトは §2.3 のアルゴリズムと同一のピクセル幅計算を
       実装しており、正確なCSSクラスを再決定できる。
       ・推奨コマンド: python3 ruby_css_verifier.py *.html --fix --margin 0.05
       ・--margin 0.05 で境界ケース（ratio が閾値から0.05以内）をスキップ
       ・バックアップは自動作成される
  (f) 新規にrubyを追加する場合、同一の (rb, rt) ペアがファイル内に
      既に存在すれば、その既存クラスをそのまま使用する。
  (g) ファイル内に前例のない全く新規の (rb, rt) ペアに限り、
      §2.3 のピクセル幅アルゴリズムでCSSクラスを決定する。

【同一ペアの統一ルール】
  (h) 同一の (rb, rt) ペアには同一のCSSクラスを割り当てる。
      同じ語根＋注釈でクラスがバラバラだと表示が不統一になる。
  (h2) ただし、大文字形と小文字形（例: Tiam と tiam）は rb テキストが
       異なるため、ピクセル幅比率が変わり、CSSクラスが異なることがある。
       この場合、rt 側の <br> の有無も変わり得る
       （例: tiam→XXS_S「その時;<br>あの時」vs Tiam→XS_S「その時;あの時」）。
       これは (h) の違反ではなく、ピクセル幅アルゴリズムの正常な結果である。

────────────────────────────────────────────────────────────────────────────────
2.3 CSSクラス決定アルゴリズム（ピクセル幅ベース）
────────────────────────────────────────────────────────────────────────────────

CSSクラスは「rt側のピクセル幅 ÷ rb側のピクセル幅」の比率（ratio）で
決定される。文字数の比率ではない。

※ このアルゴリズムは esp_replacement_json_make_module.py 内の
   output_format 関数に実装されているものと同一である。
   ruby_css_verifier.py もこれと同一のアルゴリズムを実装している。

■ ピクセル幅の計算方法

  Unicode_BMP全范围文字幅(宽)_Arial16.json を読み込み、
  文字列の各文字のピクセル幅を合計する。

  計算式:
    pixel_width(text) = Σ char_widths_dict.get(ch, 8)  （各文字 ch について）

  ※ JSONに存在しない文字はデフォルト幅 8 として扱う。

  参考値（Arial 16px、JSONからの実測値）:
    ラテン文字: a≈8.9, i≈3.6, m≈13.3, w≈11.6（文字により幅が大きく異なる）
    漢字: 概ね 16.0
    カタカナ: 概ね 16.0
    エスペラント字上符: ĉ≈8.0, ĝ≈8.9, ĥ≈8.9, ĵ≈3.6, ŝ≈8.0, ŭ≈8.9

■ 比率の計算

  ratio = pixel_width(rt文字列) ÷ pixel_width(rb文字列)

  ※ rt文字列: <br>タグを除去した純テキスト部分で計算する。
     例: 「継続<br>行為」→ 「継続行為」のピクセル幅で計算。

■ 閾値テーブル（ratio から CSSクラスへの変換）

  ┌─────────────────┬──────────┬──────────────────────────────────────┐
  │ 条件             │ クラス    │ 備考                                │
  ├─────────────────┼──────────┼──────────────────────────────────────┤
  │ ratio > 6.0     │ XXXS_S   │ <br>を2つ挿入（3行化）              │
  │ ratio > 3.0     │ XXS_S    │ <br>を1つ挿入（2行化）              │
  │ ratio > 2.25    │ XS_S     │ 1行                                 │
  │ ratio > 1.8     │ S_S      │ 1行                                 │
  │ ratio > 1.5     │ M_M      │ 1行                                 │
  │ ratio > 9/7     │ L_L      │ 1行（9/7 ≈ 1.2857）                │
  │ ratio > 1.125   │ XL_L     │ 1行（1.125 = 9/8）                 │
  │ それ以外         │ XXL_L    │ 1行                                 │
  └─────────────────┴──────────┴──────────────────────────────────────┘

  ※ 閾値は上から順に判定する（最初に該当した条件で確定）。
  ※ 閾値の正確な値:
     6（独立値）, 9/3(=3), 9/4(=2.25), 9/5(=1.8), 9/6(=1.5),
     9/7(≈1.28571), 9/8(=1.125)

■ <br>の挿入位置

  XXXS_S の場合（<br>×2）:
    rt文字列の合計ピクセル幅を3等分し、
    1/3地点と2/3地点に達した直後の文字境界に<br>を挿入する。
    （insert_br_at_third_width 相当）

  XXS_S の場合（<br>×1）:
    rt文字列の合計ピクセル幅を2等分し、
    1/2地点に達した直後の文字境界に<br>を挿入する。
    （insert_br_at_half_width 相当）

  ※ 挿入位置もピクセル幅ベースであり、文字数の等分ではない。

────────────────────────────────────────────────────────────────────────────────
2.4 AIによる新規rubyのCSS決定手順
────────────────────────────────────────────────────────────────────────────────

新たにrubyタグを追加する場合、または既存rubyの注釈テキストを
修正する場合（§2.2(e2)参照）は、以下の手順に従う。

  Step 1: 同一 (rb, rt) ペアが同一ディレクトリ内のHTMLファイルに既存か検索する
          → まず同一ファイル内を検索し、なければ他ファイルも検索する
          → 存在すれば、そのクラスをそのまま使用する（以降のStepは不要）

  Step 2: Unicode_BMP全范围文字幅(宽)_Arial16.json のピクセル幅データを
          使い、rb側・rt側それぞれの合計ピクセル幅を計算する

  Step 3: ratio = rt側ピクセル幅 ÷ rb側ピクセル幅 を求める

  Step 4: §2.3 の閾値テーブルに従ってクラスを決定する

  Step 5: ratio > 3.0 の場合はrt内に <br> を挿入する
          - ratio > 6.0 → <br>×2（ピクセル幅の1/3・2/3地点）
          - ratio > 3.0 → <br>×1（ピクセル幅の1/2地点）

  ★ Step 1 が最優先。既存ペアのクラスに合わせることが最も重要。
  ★ JSONファイルを読めない場合は、Step 1 のみで対応する。
     それでも前例のない新規ペアは、CSSクラスを空欄にしてユーザーに
     確認を求めるか、文字数比率による粗い近似値を使うが、
     「確定値ではなく近似値である」ことを必ず明記すること。

  ★ 既存CSSクラスの一括検証・修正には ruby_css_verifier.py を使用する。
     このスクリプトは上記 Step 2〜5 と同一のアルゴリズムを実装しており、
     ファイル全体のCSSクラスを正確に再計算・修正できる（§2.2(e3)参照）。

────────────────────────────────────────────────────────────────────────────────
3.1 注釈テキストの誤り（語義の間違い）
────────────────────────────────────────────────────────────────────────────────

最も重大なカテゴリ。エスペラント語根の意味を誤って日本語訳している箇所。

【典型的な誤りパターン】

(A) 類似前置詞の取り違え
    - pro（〜ゆえに）を「ために」と誤訳 ← 「ために」は por の意味
    - 修正前：<ruby>Pro<rt class="S_S">ために</rt></ruby>
    - 修正後：<ruby>Pro<rt class="S_S">〜ゆえに</rt></ruby>
    → por（目的）と pro（理由）の違いを正確に区別すること

(B) 語根の基本義の取り違え
    - paŝ（一歩）を「歩く」と誤訳 ← 「歩く」は marŝi の意味
    - 修正前：<ruby>paŝ<rt class="XL_L">歩く</rt></ruby>
    - 修正後：<ruby>paŝ<rt class="XL_L">一歩</rt></ruby>

(C) 接尾辞の基本義の揺れ
    - eg（強大）を「巨大」「非常に」と表記揺れ
    - an（成員）を「会員」と表記
    → 各接辞は1つの統一された訳語を使用すること

(D) 英語のまま残された注釈
    - dev（義務）が "must" のまま
    → 日本語注釈に統一すること

(E) HTMLタグの混入
    - neniam（決して〜ない）の注釈に「td>」が混入
    - 修正前：決して~ないtd>
    - 修正後：決して~ない
    → 注釈テキスト内にHTMLタグ残骸がないか検査すること

【よく見られる語義の誤り例】

  語根         誤            正              備考
  ─────────────────────────────────────────────────────
  pro/Pro      ために        〜ゆえに        por との混同
  paŝ          歩く          一歩            marŝi との混同
  tro          見える        過度に          完全な誤り
  rezult       起きる        結果            動詞形の誤訳
  dev          must          義務            英語が残存
  sin/Sin      懐            自身を          再帰代名詞の誤訳
  civit        自治都市      都市(国家)      語根はcivitas由来。civitano=市民は-an-で表現
  ŝarĝ        積荷          (に)荷を負わす  品詞方向の修正
  afekci       愛情          (を)感動させる  英語 affection との混同（偽の友）
  kondamn      罪            (を)断罪する    「罪」は krimo。kondamni=有罪宣告
  pane         停止          故障            「停止」は halto。paneo=故障
  proviz       貯蔵          (を)供給する    「貯蔵」は stokado。provizi=供給
  arkiv        古文書        記録保管所      arkivo=保管施設。古文書自体ではない

────────────────────────────────────────────────────────────────────────────────
3.2 ルビ未付与の単語（注釈の欠落）
────────────────────────────────────────────────────────────────────────────────

ルビなしの裸テキストとして残されたエスペラント単語を検出して注釈を付与する。

【ルビが欠落しやすい単語カテゴリ】

(A) 定冠詞 La / la
    - 文頭の "La" が裸テキストのまま放置されるケースが非常に多い
    → 全文検索で La/la の未注釈箇所を洗い出すこと

(B) 固有名詞（地名・人名・時代名）
    - Japanio（日本）, Usono（アメリカ）, Meiĵi（明治）, Edo（江戸）,
      Ŝooŭa（昭和）, Tokio（東京）, Oosako（大阪）, Nagasaki（長崎）等
    → 固有名詞は語根分解せず、固有名詞単位で1つのrubyを付与する
    → [地名]、[人名] などの分野タグを注釈に含める

(C) 国名・民族名の形容詞
    - ĉina（中国の）, japana（日本の）, nederlanda（オランダの）等
    → 国名系語根は、対応する日本語の国名をルビとする
    → japana, ĉina 等は過剰分解を避け、必要に応じて語全体を1ruby化する

(D) 一般語彙の漏れ
    - agrikulturo（農業）, erao（時代）, restaŭro（維新）等
    - eĉ（さえ）が大量に裸テキスト
    → 2文字以上のエスペラント語根はすべて注釈対象

(E) 複合語の一部
    - komputilo の "komput" 部分が裸テキストで "il" だけにルビ
    → 複合語であっても各語根に個別のルビを付与すること

(F) 略語・頭字語
    - UEA, JEI, TEJO, IJK, KEA, NRO, NPT, NPO 等
    → [略]タグ付きで日本語注釈を付与（§8.5参照）

(G) 敬称・称号・一般略語
    - s-ro, s-ino, d-ro, prof., n-ro, ktp., t.n. 等
    → [敬称]または[略]タグ付きで注釈（§8.5参照）

(H) エスペラント文中に引用された外来語
    - 英語・フランス語等の短い引用語が裸テキストで残存
    → 文脈に応じて注釈を付与するか、意図的な裸テキストとして許容するか判断

────────────────────────────────────────────────────────────────────────────────
3.3 CSSサイズクラスの不適切な割当て
────────────────────────────────────────────────────────────────────────────────

ルビの表示位置やサイズが不適切になるCSSクラスの選択ミス。

【チェックポイント】

(A) 3行注釈（<br>×2）に XXS_S を使用 → XXXS_S に修正
    - 例：「(に)荷<br>を負<br>わす」に XXS_S → XXXS_S
    - 例：「~すれ<br>ばする<br>ほど」に XXS_S → XXXS_S
    → <br>の個数を数え、適切なクラスを確認すること

(B) 1行注釈に XXS_S を使用 → XS_S等に修正
    → 1行注釈に XXS_S/XXXS_S が使われていないか全件検査

(C) 同一注釈テキストの同一語根でクラスが不統一
    → 同一の語根＋注釈の組み合わせでクラスが統一されているか確認

(D) 新規追加rubyのクラスが不適切
    → ピクセル幅比率ベースで決定されているか確認（§2.3参照）
    → ★AIの手動再計算による既存CSSの変更は禁止（§2.2(e)参照）
    → ★一括修正には ruby_css_verifier.py を使用すること（§2.2(e3)参照）

────────────────────────────────────────────────────────────────────────────────
3.4 同一語根に対する注釈の不統一
────────────────────────────────────────────────────────────────────────────────

同じ語根に異なる注釈テキストが使われている不整合。
学習者の混乱を防ぐため、可能な限り統一する。

【統一すべきケース（表記揺れ）】

  語根         揺れの例              統一先
  ─────────────────────────────────────────────────────
  eg           強大 / 巨大 / 非常に  → 強大
  eĉ           (~)さえ / さえも      → (~)さえ
  apart        別々の / 離れて       → 別々の
  san          健康な / 健康         → 健康な
  ĝeneral      全般の / 一般的な     → 全般の
  urĝ          さし迫る / 緊急に     → さし迫る
  aprior       先天的な / 先天的に   → 先天的な  ※gold辞書・アプリ準拠(先験的な も可)
  aposterior   後天的 / 後天的に     → 後天的    ※gold辞書・アプリ準拠(後験的な も可)
  do/Do        それゆえ / そこで     → それゆえ
  mekanism     機構 / メカニズム     → 機構
  Am/am        愛する / 愛           → 愛する
  demokrat     民主 / 民主主義者     → 民主
  anim         魂 / 心               → 魂

【統一すべきでないケース（正当な多義語）】

  同形異義語は文脈に応じた異なる注釈が正しい。

  語根    意味1          意味2              備考
  ──────────────────────────────────────────────────
  en      中で           円                 前置詞 vs 通貨名
  kun     (~と)共に      [日]訓読み         前置詞 vs 日本語概念
  ĉar     なぜなら       荷車               接続詞 vs 名詞語根
  radio   ラジオ         放射線             異なる語義
  sek     乾いた         [単]秒             形容詞 vs 単位
  kultur  文化           耕作する           抽象義 vs 農業義（kulturejo=耕作場）
  kod     暗号           コード             暗号義 vs 情報学義
  ĉiel    空             あらゆる方法で     名詞語根 vs 相関詞副詞
  kamp    野原           分野               物理的な野 vs 抽象的な領域
  Orden   騎士団         勲章               宗教・軍事的騎士団 vs 叙勲
  kript   暗号           地下聖堂           暗号学 vs 建築
  objekt  対象           施設               抽象的対象 vs 物理的施設
  organ   機関誌         器官               出版物 vs 身体部位
  model   模範           模型               抽象的模範 vs 物理的模型
  art     芸術           技術               創作活動 vs 実用的な技能

  lanĉ    発射           開始               物理的打上げ vs 事業開始
  vic     列             副                 物理的な列 vs 役職の副

  → 多義語の判断にはエスペラントの語源知識が必要

────────────────────────────────────────────────────────────────────────────────
3.5 改行位置（<br>）の不整合
────────────────────────────────────────────────────────────────────────────────

2行注釈における <br> の挿入位置が不統一なケース。

  語根    不統一例                   統一先
  ──────────────────────────────────────────
  ĵus     たっ<br>た今 / たった<br>今 → たった<br>今
  ia      ある種<br>の / ある<br>種の  → ある<br>種の

  → 同一注釈テキストで <br> 位置が異なる箇所を検索し統一すること
  → <br>の挿入位置はピクセル幅ベースの等分点が原則（§2.3参照）

────────────────────────────────────────────────────────────────────────────────
3.6 HTML構造上の問題
────────────────────────────────────────────────────────────────────────────────

(A) rubyタグとrtタグのバランス
    → <ruby>の開始タグ数 = </ruby>の閉じタグ数 を確認
    → <rt の開始数 = </rt> の閉じ数 を確認

(B) rtタグ内へのHTMLタグ残骸の混入
    → 「td>」等が注釈テキストに紛れ込んでいないか検査

================================================================================
4. エスペラント語学上の注意点
================================================================================

────────────────────────────────────────────────────────────────────────────────
4.1 接頭辞を含む外来語の誤分解
────────────────────────────────────────────────────────────────────────────────

エスペラントの接頭辞（mal-, re-, ek-, dis-, el-, inter-, pri- 等）と
同じ文字列で始まる外来語を誤って分解しないこと。

  【正しく1語根として扱うべき単語の例】
  element   → el + ement × （elementのelはエスペラント接頭辞el-ではない）
  interes   → inter + es × （interは接頭辞ではない）
  princip   → pri + ncip ×
  preciz    → pre + ciz ×
  prezent   → pre + zent ×
  profesi   → pro + fesi ×
  program   → pro + gram ×
  surpriz   → sur + priz ×
  kondiĉ    → kon + diĉ × （konはEの接頭辞ではない）
  konstru   → kon + stru ×
  diskriminaci → dis + kriminaci × （1語根の外来語）
  futbal    → fut(フィート) + bal(ボール) × （英語 football の借用。1語根）

  → 外来語（国際語彙）はエスペラント接頭辞で分解しない

  【語根境界の誤認識による偽分解（接頭辞以外）】
  kampus    → kamp(野原) + us(条件法) × （英語 campus の借用語。1語根）
  rokenrol  → ro + ken(樹脂材) + rol(役割) × （英語 rock'n'roll の借用語。1語根）
  ident     → i + dent(歯) × （ラテン語 idem 由来。ident が1語根）
  invest    → in + vest(服) × （invest = 投資する が1語根）
  dekstrem  → d + ekstrem(極端な) × （正: dekstr(右の) + em(傾向)）
  pupet     → pu + pet(頼む) × （正: pup(人形) + et(弱小)）

  → 接頭辞だけでなく、語根内部の偶然一致にも注意すること
  → 特に2〜3文字の語根は多くの借用語や他の語根と部分一致しやすい

  【国際語根の先頭一致による誤分解】
  Animacio  → Anim(魂) + acio × （正: Animaci(アニメーション) + o）
  brutale   → brut(粗野な) + ale × （正: brutal(残忍な) + e）
  okupaciaj → okup(占める) + aciaj × （正: okupaci(占領) + aj）
  → 国際語根の先頭部分がエスペラント固有の語根と偶然一致する場合がある
  → 単語全体の意味と文脈から正しい語根境界を判断すること

  【医学・科学用語の誤分解】
  boreliozo → borel(耳) + iozo × （ボレリア症。1語根の医学用語）
  antiseptik → anti + septik × （語根境界は正しいが、septik≠7 に注意）

  【エスペラント文中に出現する他言語の語彙】
  エスペラント文中に引用・言及される他言語（トキポナ語、サンスクリット語等）の
  語彙が、エスペラント語根として誤分解されることがある。
  sitelen  → sitel(バケツ) + en(中で) × （トキポナ語「書く;描く」。1語）
  cintamani → cinta + mani(マニア) × （サンスクリット語「如意宝珠」。1語）
  → 文脈がエスペラント以外の言語について論じている場合、
    その言語固有の語彙が語根分解の対象外であることに注意する

  【接辞境界の1文字ズレによる偽語根出現】
  自動語根分解において、接辞と文法語尾の境界が1文字分ずれ、
  偶然別のエスペラント語根と一致することがある。
  意味が完全に崩壊するため致命的なエラーとなる。

  発生メカニズム:
    正しい分解:  語根 + 接辞 + 語尾    例: neces + eg + as（非常に必要だ）
    誤った分解:  語根 + 語尾 + 偽語根   例: neces + e + gas(ガス)

  代表的な偽語根と、正しい接辞分解の対応:
    偽語根            正しい分解
    ──────────────────────────────────────────────
    tan(なめす)     ← it(受動完了) + an(対格)
    gant(手袋)      ← ig(使役) + ant(能動継続)
    gas(ガス)       ← eg(強意) + as(現在形)
    gaj(陽気な)     ← eg(強意) + aj(複数) 等
    ton(楽音)       ← at(受動継続) + on(対格)
    dis(分散)       ← ad(継続行為) + is(過去形)
    ĉas(狩る)       ← aĉ(劣悪) + as(現在形)
    ĝoj(喜ぶ)       ← iĝ(自動化) + oj(複数)
    ego(自我)       ← eg(強意) + o(名詞語尾)
    dot(持参金)     ← send+ot が sen+dot に（語根先頭の浸食）
    nest(巣)        ← en+est が e+nest に（同上）
    ret(網)         ← vir+et が vi+ret に（語根末尾の浸食）
    tul(チュール)   ← alt+ul が al+tul に（同上）

  検出方法:
    (1) ファイル内で出現回数が1〜2回の具体名詞（動物・道具・食品等）の
        注釈を抽出し、文脈的に意味が通るか検証する
    (2) </ruby>直後に裸母音（i, e, a, o）が1文字あり、その直後の
        <ruby>内の語根が文脈に合わない場合、境界ズレを疑う

  → 接頭辞の誤分解（§4.1前半）とは異なるメカニズムだが同様に致命的
  → 特に受動分詞語尾（-itan, -atan）と使役分詞語尾（-iganta）に多発

────────────────────────────────────────────────────────────────────────────────
4.2 同形異義語（多義語）の扱い
────────────────────────────────────────────────────────────────────────────────

同じ綴りで異なる意味を持つ語根がある。これらは文脈に応じて
異なる注釈を付けるのが正しい。

  ĉar → 接続詞「なぜなら」 vs 名詞語根「荷車」（ĉaro）
  en  → 前置詞「中で」 vs 名詞「円」（日本の通貨）
  dis → 接頭辞「分散」 vs 他の用法
  pas  → 動詞語根「通過する」（最も一般的な意味）
  pasi → 名詞語根「情熱」（pasio） vs 語源的要素「全」（pasigrafio内）

  → 文脈から語義を判断し、適切な注釈を選択する
  → 同形異義語は「統一すべき表記揺れ」と区別する
  → セミコロン等で訳語を無理に詰め込まず、出現ごとに適切な訳語を選ぶ

────────────────────────────────────────────────────────────────────────────────
4.3 国名・民族名の注釈方針
────────────────────────────────────────────────────────────────────────────────

国名語根の注釈には「〜人」を付けない。

  修正前        修正後       理由
  ─────────────────────────────────────
  イギリス人 → イギリス    語根はangl-（イギリスの）であり
  フランス人 → フランス    「人」はエスペラントの-anoで表現
  ドイツ人   → ドイツ      語根の基本義は国/地域名
  ロシア人   → ロシア
  スペイン人 → スペイン
  ポーランド人→ポーランド
  ヘブライ人 → ヘブライ

  → 語根の注釈は語根の基本義（国名）とし、
    「〜人」は接尾辞 -an- の注釈で表現する
  ※ 本規則は§6.2に列挙した語根(angl, franc, german, rus, hebre, hispan, pol等)に適用する。
    japan は最上位権威の2890 CSV(japano=「日本人;日本国民」)およびアプリCSVマスターに従い
    「日本人」を正とし、本規則の適用外とする(turk=トルコ人等、アプリCSVが「〜人」を正とする
    語根も同様)。

────────────────────────────────────────────────────────────────────────────────
4.4 語根の品詞的性質と注釈形式
────────────────────────────────────────────────────────────────────────────────

エスペラントでは語根に文法語尾（-o, -a, -e, -i, -as等）を付けて
品詞を変える。ルビ注釈は語根の基本義を1つの形式で統一する。

  原則：同一語根には文脈によらず同一注釈テキストを使用する
        （同一ディレクトリ内の全ファイルを通じて統一する。多義語を除く）

  例：san → 常に「健康な」（「健康」「健康に」等と揺らさない）
      apart → 常に「別々の」（「離れて」等と揺らさない）
      aprior → 常に「先験的な」（「先験的に」と揺らさない）

  理由：文法語尾（-a, -o, -e等）がそれぞれ「形容詞」「名詞」「副詞」
        という品詞情報を持っているため、語根側の注釈は基本形で統一する

================================================================================
5. 修正作業の具体的手順
================================================================================

以下の手順で、効率的かつ網羅的に修正を進めることを推奨する。

────────────────────────────────────────────────────────────────────────────────
第1段階：全ルビペアの抽出と一貫性検査（最優先）
────────────────────────────────────────────────────────────────────────────────

  (1) ファイルからすべての <ruby>語根<rt class="X">注釈</rt> ペアを抽出
  (2) 同一語根に対して異なる注釈がある箇所をリストアップ
  (3) 各ケースを「表記揺れ（統一すべき）」か「多義語（維持すべき）」に分類
  (4) 表記揺れの場合、多数派の注釈を正とし、少数派を修正

────────────────────────────────────────────────────────────────────────────────
第2段階：CSSクラスの適切性検査
────────────────────────────────────────────────────────────────────────────────

  (1) <br>を2つ含むrtにXXXS_S以外が使われていないか検査
  (2) <br>を1つ含むrtにXXS_S以外が使われていないか検査
  (3) <br>を含まないrtにXXXS_SまたはXXS_Sが使われていないか検査
  (4) 同一語根＋注釈でクラスが不統一でないか検査
  (5) ruby_css_verifier.py でピクセル幅比率ベースの一括検証を実行
      → python3 ruby_css_verifier.py *.html（検証のみ、修正なし）
      → 修正適用: python3 ruby_css_verifier.py *.html --fix --margin 0.05
  ★ AIの手動再計算による既存CSS変更は禁止（§2.2(e)参照）
  ★ スクリプトによる一括修正は推奨（§2.2(e3)参照）

────────────────────────────────────────────────────────────────────────────────
第3段階：ルビ未付与語の検出
────────────────────────────────────────────────────────────────────────────────

  (1) </ruby>タグと<ruby>タグの間に2文字以上のアルファベット列がないか検査
      ※ 3文字以上だけでなく、2文字の独立語（mi, vi, li, ni, ne, du, ok 等）も
        頻繁に漏れるため、2文字から検査する
  (2) 特に定冠詞 La/la（省略形 l' を含む）の未注釈箇所を重点的に検査
  (3) 固有名詞（地名・人名）の未注釈箇所を検査
  (4) 人称代名詞・小詞・相関語の未注釈箇所を検査
      （§9「必ずruby化すべきもの」(4)-(6b) の全語を対象とする）
  (4b) <br>直後や引用符直後（対話行頭）の裸テキストを重点的に検査
       ※ "Se ...", "Ĉu ...", "Nu, ..." 等の対話行頭パターンは
         ruby間の接合部とは異なる位置に出現するため、別途検査が必要

────────────────────────────────────────────────────────────────────────────────
第4段階：注釈テキストの正確性検査（エスペラント知識が必要）
────────────────────────────────────────────────────────────────────────────────

  (1) 前置詞の意味が正しいか（por/pro/per/pri等の区別）
  (2) 接頭辞・接尾辞の基本義が正しいか
  (3) 外来語が誤って語根分解されていないか
  (4) 注釈内にHTMLタグの残骸が混入していないか

────────────────────────────────────────────────────────────────────────────────
第5段階：改行位置の統一
────────────────────────────────────────────────────────────────────────────────

  (1) 2行注釈で同一注釈テキストの<br>位置が不統一な箇所を検出
  (2) 同一 (rb, rt) ペアの既存 <br> 位置に統一する

────────────────────────────────────────────────────────────────────────────────
第6段階：HTML構造の検証
────────────────────────────────────────────────────────────────────────────────

  (1) ruby/rtタグのバランス（開始・終了タグの数が一致）
  (2) 全体の整合性確認

6. 固有名詞・国名の方針
================================================================================

────────────────────────────────────────────────────────────────────────────────
6.1 固有名詞の分類タグ
────────────────────────────────────────────────────────────────────────────────

固有名詞には以下のカテゴリタグを付けて注釈する。

■ 使用するタグ一覧
  [人名]    : 個人名（日本人・外国人を問わず）
  [地名]    : 国名・都市名・地域名
  [宗教]    : 宗教関連の固有名詞（※ 一般的な宗教用語の分野タグは [宗]。§7.6参照）
  [架空国]  : 架空の国や地域
  [雑誌]    : 雑誌・刊行物の名称
  [言語名]  : 言語の名称
  [団体]    : 組織・団体名
  [日]      : 日本語の概念（音読み/訓読み等）
  [語]      : エスペラント文中に引用された他言語の語彙・概念
  [書]      : 書籍・著作物の名称（[雑誌]とは区別）
  ※ 略語・敬称には [略]・[敬称] 等のタグを使用する（§8.5参照）
  ※ 上記以外の分類タグ・分野タグを含む完全な正規一覧と統合ルールは §7.7 を参照

■ 注釈形式の例
  <ruby>Edmond Privat<rt class="S_S">[人名]エドモン・プリヴァ</rt></ruby>
  <ruby>Antananarivo<rt class="M_M">[地名]アンタナナリボ</rt></ruby>
  <ruby>La Revuo Orienta<rt class="S_S">[雑誌]ラ・レビューオ・オリエンタ</rt></ruby>
  <ruby>Tokipono<rt class="S_S">[言語名]トキポナ</rt></ruby>
  <ruby>Academia Vivarium Novum<rt class="M_M">[団体]アカデミア・ヴィヴァリウム・ノヴム</rt></ruby>

■ ルール
  → 固有名詞は必ず上記のカテゴリタグ付きで注釈すること
  → 人名はフルネーム単位で1つのrubyにまとめる
  → 日本人名は、漢字表記が確定している場合は rt に漢字を記載する
  → 仮名名義・芸名・ハンドル名など漢字が確定しない場合は、
    仮名等の実表記を用いる
  → 近接する日本語訳・キャプション・原注に正規表記がある場合は、
    それを最優先で採用する

────────────────────────────────────────────────────────────────────────────────
6.2 国名・民族名の注釈統一
────────────────────────────────────────────────────────────────────────────────

■ 変更すべき語根（「〜人」→ 国名）
  Angl / angl : イギリス人 → イギリス
  Franc / franc: フランス人 → フランス
  German / german: ドイツ人 → ドイツ
  Rus / rus   : ロシア人 → ロシア
  hebre       : ヘブライ人 → ヘブライ
  hispan      : スペイン人 → スペイン
  pol         : ポーランド人 → ポーランド

■ 理由
  エスペラントでは語根は国名を示し、「〇〇人」は接尾辞 -ano で表す。
  例: angl → イギリス、anglano → イギリス人
  従って語根そのものの注釈は「イギリス」が正しい。

■ ルール
  → 上記列挙の語根については、国名語根の注釈に「人」を付けない
    (japan・turk 等、2890 CSV・アプリCSVマスターが「〜人」を正とする語根は対象外)
  → 「〇〇人」と書くべきは rb が「国名語根+an」
    （例: anglano, francanoj など）の場合のみ
  → 国名・地名語根への [地名] タグは任意とする。
    カタカナ表記で地名と自明な場合（ポーランド、フランス等）は省略してよい。

────────────────────────────────────────────────────────────────────────────────
6.3 固有名詞の誤分解防止
────────────────────────────────────────────────────────────────────────────────

固有名詞をエスペラント語根として誤って分解しないこと。

■ 誤分解の典型例
  Leon   : [動]ライオン → 実際は人名の一部
  Privat : 私的な → 実際は人名 Edmond Privat の一部
  Stal   : 馬小屋 → 実際は人名 Stalin の一部
  Juli   : 7月 → 実際は人名 Julio (Baghy) の一部
  Hok    : 鉤 → 実際は地名 Hokkaido の一部
  Ok     : 八 → 実際は人名 Oka の一部
  Sak    : 袋 → 実際は人名 Sakai の一部

■ 韓国人名・英語タイトルの誤分解例
  Jun    : 若い → 実際は韓国人名 Lee Jun-seok の一部
  Suk    : 汁 → 実際は韓国人名 Yoon Suk Yeol の一部
  Kuk    : ケーキ → 実際は姓 John Kuk
  Mi     : 私 → 実際は訓民正音（Hun Min Jeong Eum）の一部
  Gam    : 音階 → 実際は英語タイトル Squid Game の一部

■ 正しい処理
  → 固有名詞はフルネーム（またはエスペラント表記の単位）で1つの ruby に収める
  → エスペラント語根と同形の人名・地名は分解しない
  → 例: <ruby>Edmond Privat<rt class="S_S">[人名]エドモン・プリヴァ</rt></ruby>
        （「Privat」を「私的な」と分解しない）

■ 実例集: アプリ全数すり合わせ(2026-07, 全123文書×7ラウンド)で検出・是正済みの誤分解
  以下は本コーパスで実際に発生していた固有名詞の偽語根。同型の再発防止の参考とすること。
  Klara     : Klar(明瞭な)+a × → [人名]クララ（ザメンホフ夫人 Klara Zilbernik）
  Chiune    : Ch+iu(ある人)+ne × → [人名]千畝（杉原千畝）
  Sugihara  : Sugi+har(毛)+a × → [人名]杉原
  Fukui     : Fuk(ヒバマタ)+ui × → [人名]福井 ／ Masaharu: Ma+sahar(サハラ)+u × → [人名]正晴
  York      : Y+ork(シャチ) × → [地名]ヨーク（York大学）
  Ikeda     : Ik+eda(エッダ) × → [地名]池田
  Okayama   : Ok(八)+ay+am(愛する)+a × → [地名]岡山
  Fogel     : Fo+gel(霜) × → [人名]フォーゲル
  Nemunas   : N+em(傾向)+u+nas(うけ) × → [地名]ネムナス（カウナスの川）
  Lusin     : Lu(賃借する)+si(自分)+n × → [人名]魯迅
  Antoni    : Anton+i（-iを動詞語尾扱い） × → [人名]アントニ（ポーランド語名 Jan Antoni）
  Tiba      : Tib+a（-aを形容詞語尾扱い） × → [地名]千葉（gubernio Tiba＝名詞。語末-aは名前の一部）
  Barato系  : bar(障害)+at(受動継続)+a × → barat(インド)+a（国名Barato の形容詞 barata）
  ※ 借用語の同型例: platformo を plat(平たい)+form(形)+o と割らない（1語根 platform）。
  ※ 名前の語末が -i / -a / -o でも、文脈が名詞（前置詞の目的語・同格）なら語尾分解しないこと
    （Antoni・Tiba 型）。逆に Blis/a（ブリス文字）・Kanto/a（関東+形容詞語尾）のように
    実際に形容詞用法の場合は分離が正しい。判断は必ず対訳・文脈で行う。

■ ラテン語由来の人名の誤分解
  ラテン語由来の人名はエスペラント語根の組合せに見えることがある。
  Urbano  → Urb(市) + an(成員) + o × （教皇ウルバヌス。人名）
  → 歴史的人名（教皇名・古典人名等）は語根分解せず1語で扱うこと

■ ポーランド語・スラブ系固有名詞の誤分解
  ポーランド語やスラブ系の地名・人名は、エスペラント語根と部分一致しやすく、
  特に誤分解の頻度が高い。
  Wielkopolska → Wiel + ko + pol(ポーランド) + ska × （ヴィエルコポルスカ地方。1語）
  → ポーランド語の地名・人名を見たら、まず1語の固有名詞を疑うこと

■ 日本語固有名詞（非標準ローマ字表記）の誤分解
  日本語の人名・地名がエスペラント式ローマ字で表記される場合、
  語根と偶然一致する部分が生じやすい。
  Kameoka  → Kame(美) + oka × （亀岡。1語の地名）
  Hidemaru → ide(観念) + mar(海) + u × （秀丸。1語の人名）
  Timpuo   → Tim(恐れる) + puo × （ティンプー。首都名）
  → 日本の人名・地名・宗教用語がエスペラント式で表記された場合は特に注意

■ 検出のヒント
  → 大文字で始まる語が連続する場合は固有名詞の可能性を疑う
  → 特に日本人名・韓国人名のローマ字表記はエスペラント語根と偶然一致しやすい
  → ポーランド語・チェコ語等のスラブ系固有名詞も偶然一致が多い
  → 英語のタイトル名（TV番組名等）の一部がエスペラント語根と一致する場合もある
  → 文脈が特定の言語・文化について論じている場合、その言語の固有語彙に注意する

7.6 丸括弧の用法ルール
────────────────────────────────────────────────────────────────────────────────

■ 格情報（動詞の格支配を示す）
  (を)     : 他動詞の対格を示す      ← 圧倒的最多
  (に)     : 間接目的語・方向
  (の)     : 連体修飾
  (~と)    : 共格・対象
  (~も)    : 包含
  (~の)    : 連体修飾の別形
  (~から)  : 起点

■ 補足情報（語義の限定）
  (~)      : チルダで不定の目的語
  (達)     : 複数を示す
  (時)     : 時間文脈
  (所有)   : 所有文脈
  (尊称)   : 敬称
  (暦の)   : 暦文脈

■ 用法ルール
  → 格情報は必ず丸括弧で表記する：(を), (に), (~と), (~の)
  → 分野限定は角括弧で表記する。主な分野タグ：
    [化], [理], [植], [軍], [楽], [貨], [文], [宗], [医], [解], [地], [政],
    [動], [法], [史], [語], [虫], [商], [劇], [単], [数], [料], [気], [果],
    [農], [鳥], [建], [機], [生], [学], [梵]
    （上記は代表例。分野タグ・分類タグの完全な正規一覧と「揺れ→正規」統合ルールは
      §7.7 を参照。新規タグの考案は §7.7 の運用原則に従うこと）
  → 丸括弧と角括弧を混同しない
  → チルダ~ は目的語・補語の省略を示す: (~)なしで = 「~なしで」

  [書名]→[書]     [宗教名]→[宗教]    [民族名]→[民族]    [病名]→[病]

================================================================================
8. 句読点・語尾・複合語・記号ルール
================================================================================

────────────────────────────────────────────────────────────────────────────────
8.1 句読点の配置ルール
────────────────────────────────────────────────────────────────────────────────

【原則】句読点はrubyタグの外側に、文法語尾のさらに外側に置く。
  rubyの中（rb内）には句読点を含めない。

  1. カンマ (,)     → </ruby>語尾, の形。rb内に含めない
  2. ピリオド (.)   → </ruby>語尾. の形。rb内に含めない
  3. 疑問符 (?)     → </ruby>語尾? の形
  4. 感嘆符 (!)     → </ruby>語尾! の形
  5. セミコロン (;) → </ruby>; の形（文語尾の直後）
  6. コロン (:)     → </ruby>: の形
  7. 例外: 略語のピリオド (k.t.p., t.e., ekz.) はrb内に含める

────────────────────────────────────────────────────────────────────────────────
8.2 文法語尾の切断位置ルール
────────────────────────────────────────────────────────────────────────────────

【原則】語根部分のみをrubyのrb内に含め、文法語尾は裸テキストとして
ruby直後に配置する。

  正当な語尾パターン（17種）:
    名詞系: o, oj, on, ojn
    形容詞系: a, aj, an, ajn
    副詞・不変化語: e, en
    動詞不定・命令: i, u
    対格・複数: n, j, jn
    特殊: io, ia
      （主に国名形成: Japan+io=Japanio 等の語根直後の語尾として出現する場合）
      ※ io, ia に対格 -n や複数 -j が後続した複合語尾
        （ion, ian, ioj, iojn, iaj, iajn）も語根直後であれば正当な裸テキストである
      ★ 独立した相関語 io（何か）, ia（ある種の）およびその屈折形
        （ion, ian, ioj, iajn 等）は独立語であり、§7.5に従い全体で1 rubyとする。
        文脈で「語根の語尾」か「独立相関語」かを判別すること

  ※ as/is/os/us は大半が独立ruby（例: <ruby>as<rt class="XS_S">現在形</rt></ruby>）で表現される。
  ※ ruby直後に単独の "s" が出現した場合は、語根分割の誤りを疑うこと。

  語根末尾の音韻ルール:
    - rb末尾が子音で終わる → 標準パターン（約73%）
    - rb末尾が母音で終わる → 以下のカテゴリのみ許容（約27%）:
      (1) 機能語: la, de, mi, ne, ni, ke, ĝi, li, vi, si, ili, pri, da 等
      (2) 相関語: kiu, tiu, ĉiu, tio, kio 等
      (3) 数詞: unu, du 等
      (4) 母音末尾語根（例外的）: naci, ali, pli 等

  誤分解の検出指標:
    ruby直後に非標準語尾パターン（ek, toj, ante, saja, bergo, aĵo等）が
    出現したら、語根境界の誤認識の可能性が高い。

────────────────────────────────────────────────────────────────────────────────
8.3 複合語の分解境界ルール
────────────────────────────────────────────────────────────────────────────────

【原則】複合語（多語根語）は語根単位に分解し、各語根を独立rubyとする。
語根間の接合母音（通常 o）は裸テキストとして配置する。

  接合部の正当パターン:
    (1) 文法語尾のみ: o, a, e, i, u, oj, aj, on, an, en 等
    (2) 文法語尾+句読点: o,  oj,  on,  aj,  a,  等
    (3) ハイフン: - （複合語中のハイフン結合）
    (4) 引用符: " " （引用の開始・終了）

  接合部の異常パターン（修正すべき）:
    (A) 独立語の漏出: ĉi, eĉ, mi, ni, ĝi, vi 等が裸テキスト
        → 正しくはruby化する
    (B) 固有名詞の断片化: japana, ĉina, Japanio, Tokio 等
        → 正しくは全体をruby化する
    (C) 語尾の過剰切断: s, k, ek, toj, ante 等
        → 語根境界を正しい位置で切る

────────────────────────────────────────────────────────────────────────────────
8.4 引用符・ハイフンの処理ルール
────────────────────────────────────────────────────────────────────────────────

  引用符:
    - 引用符はrubyの外側に配置する（rb内に引用符を含めない）

  ハイフン:
    - 一般語根のハイフンは裸テキスト
    - 以下はハイフンごとrb内に含める:
      敬称: s-ro（氏）, s-ino（夫人）, d-ro（博士）, n-ro（番号）
      固有名詞: Taigyaku-jiken, Suihei-ŝa, Buraku-min 等

  URL・ドメイン:
    - href属性値、URL文字列、Markdown風リンク表示文字列は
      ruby化・除去しない
    - URL内部に誤ってrubyが混入している場合は、
      リンク先・表示文字列とも素のURLに戻す

────────────────────────────────────────────────────────────────────────────────
8.5 略語の処理ルール
────────────────────────────────────────────────────────────────────────────────

  1. 略語はピリオドごと全体を1つのrubyのrb内に含める
  2. 略語の注釈は日本語を優先:
     ktp. → "~など"、t.e. → "すなわち"、ekz. → "例えば"、ia. → "とりわけ"、
     t.n. → "いわゆる"
  3. 頭字語（UEA, JEI, TEJO, KEA 等）は[略]タグ付きで注釈:
     UEA → "[略]世エス協"、JEI → "[略]日エス協"、TEJO → "[略]青年組織"
  4. 敬称・称号略語は目的に応じたタグで注釈:
     s-ro → "[敬称]氏"、s-ino → "[敬称]女史"、d-ro → "[略]博士"、
     prof. → "[略]教授"
  ※ アプリCSVマスターの略語訳はタグ無し・末尾ピリオド形式(例: ekz.→例., t.e.→つまり,
    UEA→世エス協, prof.→教授.)。文言・タグ有無の差は両様とも正当であり、アプリ生成ルビを
    本節の例示に機械的に合わせる書き換えは行わないこと。
  5. 展開形を残す場合もある: k.t.p. → "kaj tiel plu"
  6. 人名の略記（L.L.Zamenhof等）もrb内にまとめる

9. 裸テキストの許容基準
================================================================================

rubyタグ外に残存するアルファベットテキストの分類基準。

■ 裸テキストとして正常なもの
  (1) 文法語尾: o, a, e, i, n, j およびその組み合わせ
      （oj, on, aj, an, ojn, ajn, en, u, jn, io, ia）
      ※ io, ia およびその屈折形（ion, ian, ioj, iojn, iaj, iajn）は、
        語根直後の文法語尾としてのみ正常な裸テキストとなる
        （例: Japan+io → Japanio）。
        独立した相関語として出現する io/ia 等は裸テキストではなく
        ruby化すべきもの（下記(6b)参照）。§8.2も参照
      ※ 序数・年代表記（42-a, 1960-aj, 1-an 等）も正常な裸テキスト
      ※ 任意複数表記 a(j), o(j) 等（辞書的・教育的文脈）も正常な裸テキスト

■ 必ずruby化すべきもの
  (2) 略語・頭字語・敬称略語: JEI, TEJO, UEA, UK, ktp., s-ro 等（§8.5参照）
  (3) 冠詞: La / la（省略形 l' を含む）
  (4) 人称代名詞: mi, li, ŝi, ĝi, vi, ni, ili, oni, si（文頭大文字形も同様）
  (5) 副詞類・小詞: eĉ, do, ne, nu, ja, ĉu, ke, se
  (6) 指示詞: ĉi
  (6b) 相関語およびその屈折形: io, ion, iu, iun, ia, ie, iel, iam, iom, ial
       およびtiu系・kiu系・ĉiu系・neniu系の全形（§7.5参照）
       ※ これらは§8.2の「正当な語尾 io/ia」と綴りが重なるが、
         独立語として出現している場合は必ずruby化する
  (7) 国名系語・固有名詞: japana, Japanio, Tokio, Meiĵi 等
  (8) その他の2文字以上のエスペラント語根

■ バグとして除去すべきもの
  (9) CSS/HTML断片・壊れたURL断片: important, size, font, em, px,
      td>, /span>, http（URLの一部だけが壊れて露出したもの）等
      → 裸テキストとして本文に混入していたら即座に除去
      → ただし正当なURL/ドメイン（https://..., x.com,
        vivariumnovum.net 等）は除去しない

10. 体系的修正パターン
================================================================================

AI修正で繰り返し適用される修正パターンを体系化する。

────────────────────────────────────────────────────────────────────────────────
【パターン1: 過剰分解の統合】（最重要・最頻出）
────────────────────────────────────────────────────────────────────────────────

  検出: 固有名詞・国名形容詞・分解不可能な外来語が語根分解されている箇所
  修正:
    - 固有名詞の語根分解を中止し、フルネームで1 ruby化
    - 国名形容詞(japana, ĉina等)の語根分解を中止
    - 分解不可能な外来語(galaksio等)の語根分解を中止

────────────────────────────────────────────────────────────────────────────────
【パターン2: 独立語の裸テキスト漏出修正】
────────────────────────────────────────────────────────────────────────────────

  検出: ruby間の接合文字列、または<br>直後・引用符直後に独立語が出現
  対象語: ĉi, eĉ, ne, ja, la, mi, li, ŝi, vi, ni, ĝi, ili, oni, si,
          ĉu, nu, ke, se, do,
          io（何か）, iu（ある人）, l'（laの省略形）,
          du, ok 等の数詞,
          および上記の大文字形（文頭）・屈折形（ion, iun, ŝin 等）
  修正: 裸テキストの独立語をruby化する
  注意: io/Io は文頭(大文字)・文中(小文字)の両方で漏れやすい
  注意: 対話形式の文書では <br> 直後や引用符 " 直後に
        大文字の独立語（"Ĉu ...", "Se ...", "Nu, ..." 等）が裸で出現しやすい。
        ruby間の接合部だけでなく、行頭パターンも検査対象とすること

────────────────────────────────────────────────────────────────────────────────
【パターン3: 語尾誤切断の修正】
────────────────────────────────────────────────────────────────────────────────

  検出: ruby直後に非標準語尾パターンが出現
         （17種の正当語尾以外: ek, toj, ante, saja, bergo, aĵo 等）
  修正: 語根境界を正しい位置に移動し、語尾を標準パターンにする

────────────────────────────────────────────────────────────────────────────────
【パターン4: 注釈の意味改善】
────────────────────────────────────────────────────────────────────────────────

  検出: 同一語根に複数の異なる注釈が混在
  修正:
    - 国名語根から「人」を除去（angl: イギリス人→イギリス）
    - 多義語の文脈別訳語選択（ĉar: だから;車 → なぜなら/荷車）
    - 分野タグの追加（[化][植][虫][解][天][建] 等）

────────────────────────────────────────────────────────────────────────────────
【パターン5: HTML構造の正規化】
────────────────────────────────────────────────────────────────────────────────

  修正:
    - CSSクラス残滓（裸テキスト中の class= 等）を除去
    - rt内のHTML残骸（td> 等）を除去

────────────────────────────────────────────────────────────────────────────────
【パターン6: 接辞境界ズレの修正】（★致命的・高頻度）
────────────────────────────────────────────────────────────────────────────────

  検出: ファイル内で1〜2回しか出現しない具体名詞の注釈
        （なめす、手袋、ガス、5月、楽音、槍騎兵 等）
  確認: 前後の裸テキスト（i, e, a, o）と合わせて読み直し、
        正しい接辞＋語尾の組み合わせにならないか検証する
  修正: ruby境界を正しい接辞境界に移動する
  注意:
    - rb内のテキスト（エスペラント原文）そのものは絶対に変更しない
    - 境界移動後の (rb, rt) ペアのCSSクラスは、同一ファイル内の
      既存ペアに合わせる（§2.2(f)参照）
    - 境界移動後にruby/rtタグのバランスが崩れていないか必ず検証する
    - 一括置換で修正する際は、rbとrtの間に改行を含むパターンが
      漏れやすい。修正後に grep -c で残存件数を確認すること
  補助的検出方法:
    (3) 同一ファイル内で同じエスペラント単語が正しく分解されている箇所と
        照合する。例: futbal が4箇所で1語根、1箇所だけ fut+bal なら後者が誤り
    (4) 参照可能なファイル（完成版・他言語版等）がある場合、(rb, rt)ペアの
        差分で偽語根を検出できる。ただし参照ファイル自身にも誤りがあり得るため、
        差分の各項目は必ず文脈に照らして正誤を判断すること
  詳細: §4.1「接辞境界の1文字ズレによる偽語根出現」を参照

12. 統合チェックリスト
================================================================================

修正完了後に以下の全項目を最終確認する。

────────────────────────────────────────────────────────────────────────────────
【A. 構造・書式】
────────────────────────────────────────────────────────────────────────────────
  □ 全rubyタグのバランスが取れている（開始数 = 終了数）
  □ 全rtタグのバランスが取れている
  □ 注釈テキスト内にHTMLタグの残骸がない

────────────────────────────────────────────────────────────────────────────────
【B. CSSクラス】
────────────────────────────────────────────────────────────────────────────────
  □ 3行注釈（<br>×2）はすべてXXXS_Sクラスを使用している
  □ 2行注釈（<br>×1）はすべてXXS_Sクラスを使用している
  □ 1行注釈にXXXS_S/XXS_Sが使用されていない
  □ 同一(rb,rt)ペアに同一クラスが割り当てられている
  □ AIが手動で既存CSSクラスを変更していない（§2.2(e)）
  □ 新規rubyのCSSクラスがピクセル幅比率で正しく決定されている
  □ ruby_css_verifier.py で一括検証を実行済み（§2.2(e3)）

────────────────────────────────────────────────────────────────────────────────
【C. 語根分解】
────────────────────────────────────────────────────────────────────────────────
  □ エスペラントの語は「語根+接辞+語尾」に分解し、各部品を独立rubyにしている
  □ 文法語尾（o,a,e,i,n,j等）は裸テキストとして残している
      （※ as/is/os/us は独立rubyとするのが標準 → §7.3参照）
  □ 造語接辞は独立rubyとし、注釈は §7.1/7.2 の辞書に従っている
  □ 前置詞・接続詞・副詞は独立rubyとし、注釈は §7.4 の辞書に従っている
  □ 相関語は語全体で1つのrubyとし、注釈は §7.5 の辞書に従っている
  □ 外来語が誤って語根分解されていない

────────────────────────────────────────────────────────────────────────────────
【D. 固有名詞】
────────────────────────────────────────────────────────────────────────────────
  □ 固有名詞はエスペラント語根として分解していない
  □ フルネームまたは固有名詞単位で1つのrubyにまとめ、
    [人名][地名][団体][言語名][雑誌]等のタグを付けている
  □ 国名語根の注釈に「人」が付いていない（angl=イギリス, not イギリス人）

────────────────────────────────────────────────────────────────────────────────
【E. 注釈の質】
────────────────────────────────────────────────────────────────────────────────
  □ 同一語根に対する注釈テキストが統一されている（多義語を除く）
  □ 接辞の注釈は辞書に従い文書全体で統一されている
  □ 多義語はセミコロン詰め込みではなく、文脈ごとに適切な訳語を選んでいる
  □ 丸括弧は格情報に、角括弧は分野タグに使い分けている
  □ 前置詞（por/pro/per/pri等）の語義が正しい
  □ チルダの全角/半角が既存慣例と統一されている

────────────────────────────────────────────────────────────────────────────────
【F. 裸テキスト】
────────────────────────────────────────────────────────────────────────────────
  □ 定冠詞La/laがすべて注釈付きである
  □ eĉ, do, ne, nu, ja等の小詞がすべて注釈付きである
  □ 固有名詞に適切な注釈（[地名]、[人名]、[団体]、[言語名]、[雑誌]等）がある
  □ 文法語尾と略語以外のアルファベットが裸テキストで残っていない

────────────────────────────────────────────────────────────────────────────────
【G. 原文テキストの保全（★★★ 絶対厳守）】
────────────────────────────────────────────────────────────────────────────────
  □ rubyタグ内のrb（底本テキスト）の文字列を一切変更していない
  □ 語根境界の移動はrubyタグの分割・統合のみで行い、
    文字の挿入・削除・並べ替えは行っていない
  □ 原文に誤植が疑われる場合も修正せず、コメントで報告にとどめている
  □ 修正後の全rubyのrb部分を連結した文字列が、修正前と完全一致する
  □ CSS/HTML断片が本文に混入していない
  □ URL・ドメイン・リンク表示文字列に誤ってrubyが混入していない
  □ 改行位置（<br>）が同一注釈内で統一されている

────────────────────────────────────────────────────────────────────────────────
【H. 句読点・記号】
────────────────────────────────────────────────────────────────────────────────
  □ カンマ・ピリオド・疑問符・感嘆符がruby外側にある
  □ 略語（ktp., t.e., ekz.）はピリオドごとrb内にある
  □ 引用符がrb内に含まれていない
  □ ハイフンが敬称(s-ro等)・固有名詞以外でrb内に含まれていない

■ 同形異義語(§3.4末尾表)の主義選択について
  アプリは gold辞書(世界语全部单词…学習者版)の【第一義】を主注釈(CSV)に採用する。
  ガイド表の「意味1/意味2」はいずれも正当だが、どちらを主とするかは gold第一義に従う:
    kod=法典(第一義。暗号/コードは第二義以降)   art=技術(第一義。芸術は第二義)
    objekt=物体(対象は第二義)                  organ=器官(機関誌は第二義)
    orden=勲章(騎士団は【史】義)               model=模型/模範   radio: radi=光線 / radio=ラジオ
  → 文脈により第二義が正しい箇所は word_anno(文脈別注釈)で個別上書きされる。

■ gold準拠に整合した基本義(§3.1/§3.4)

    aprior = 先天的な / aposterior = 後天的  ※gold辞書の訳(先験的/後験的も可)
